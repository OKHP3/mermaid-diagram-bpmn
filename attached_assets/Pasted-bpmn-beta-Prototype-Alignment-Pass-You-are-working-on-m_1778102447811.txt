## Phase 3: Pools and lanes

This is the largest single piece of work in this prompt. Without pools and lanes, the prototype cannot honestly claim BPMN Descriptive Conformance subset coverage.

### 3.1 DSL support

Extend `bpmn-parser.ts` to recognize block syntax:

```
pool buyer "Buyer" {
  lane req "Requester" {
    start s1 "Need Identified"
    task:user t1 "Submit PO Request"
  }
  lane mgr "Manager" {
    task:user t2 "Review Request"
    xor g1 "Approved?"
    end e2 "Rejected"
  }
  s1 --> t1
  t1 --> t2
  t2 --> g1
  g1 --> t3: "yes"
  g1 --> e2: "no"
}

pool supplier "Supplier" {
  task:receive t3 "Receive PO"
  task t4 "Acknowledge PO"
  end e1 "PO Acknowledged"
  t3 --> t4
  t4 --> e1
}

%% Cross-pool message flows declared at top level outside pool blocks
t2 ~~> t3
```

Rules:
- A pool block contains lane blocks and flow declarations.
- A lane block contains node declarations only.
- Sequence flows (`-->`, `==>`) must have both endpoints in the same pool. Parser errors with a clear message if not.
- Message flows (`~~>`) cross pool boundaries and must be declared at the top level, not inside any pool block. Parser errors with a clear message if not.
- Lanes have one level of nesting only. Reject nested `lane lane`.

The hand-written parser will need to track block context. A simple stack-based approach is fine: when you see `{`, push the current scope (pool or lane); when you see `}`, pop.

### 3.2 Renderer support

Update `bpmn-renderer.tsx` to draw:
- Pool: outer rectangle wrapping all child nodes; pool label as rotated 90° text on the left edge.
- Lane: horizontal sub-rectangle inside its pool; lane label as text on the left edge (not rotated).
- Pools stack vertically with a small gap.
- Sequence flows route between nodes inside the same pool. Don't cross pool boundaries.
- Message flows route between nodes in different pools, drawn dashed with open circle source and open arrow target.

For routing inside a lane, keep the existing topological left-to-right approach. For routing across pools (message flows), connect the source node's right edge to the target node's left edge with an orthogonal path.

### 3.3 Layout updates

`bpmn-layout.ts` needs to know about lane and pool containment:
- Compute layout independently within each lane.
- Pool height is the sum of its lanes' heights.
- Total diagram height is the sum of pool heights plus gaps.
- Pool width is the maximum width across all its lanes.

Edit `pages/DslReference.tsx` to include the pool and lane block syntax with the example above. Do not delete existing reference content; add the new section in a logical place.
