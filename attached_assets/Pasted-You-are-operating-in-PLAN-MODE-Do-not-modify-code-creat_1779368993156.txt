## 2. Required Operating Rules

Follow these rules strictly.

### Do Not Hallucinate
If a detail cannot be verified, label it as an inference, assumption, or open question.

### Use Evidence Labels
Every major finding should be classified as one of:

- Verified: directly supported by files, behavior, docs, commits, issues, or visible artifacts.
- Inferred: strongly suggested by implementation or repeated patterns.
- Speculative: plausible, but needs confirmation.
- Unknown: not enough evidence.

### Preserve Uncertainty
Do not rewrite messy prototype history into fake clean project history. The point is to recover the real shape of the work, not pretend it was planned perfectly.

### Separate As-Built From Proposed
Keep current completed functionality separate from future recommendations.

Use these labels:

- AS-BUILT = what exists now.
- IMPLIED = what the current implementation suggests.
- GAP = what is missing or incomplete.
- RECOMMENDED = what should happen next.

### Plan Mode Only
Do not implement. Produce the analysis and a proposed documentation plan only.

## 5. Deliverable 3 — Product Archaeology Ledger

Create a Product Archaeology Ledger.

This is the core reconstruction table. Use the current Repl and any linked artifacts to infer completed work.

Use this format:

| ID | Evidence Source | Observed Artifact / Behavior | Inferred Completed Task | Resulting Capability | Derived Requirement | Confidence | Follow-Up |
|---|---|---|---|---|---|---|---|

Guidance:

- Evidence Source should cite the file, folder, visible feature, documentation artifact, GitHub issue, commit, PR, Notion page, or deployment behavior when available.
- Observed Artifact / Behavior should describe what is actually present.
- Inferred Completed Task should translate that evidence into work performed.
- Resulting Capability should describe what the product can now do.
- Derived Requirement should rewrite the completed capability as a retrospective requirement.
- Confidence should be High, Medium, or Low.
- Follow-Up should identify validation, cleanup, or next work.

Example requirement style:

> The system shall allow users to create, edit, and persist project records.

Do not over-infer. If the evidence is weak, label confidence as Low.
